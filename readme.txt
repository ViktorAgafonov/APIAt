# APIAt v0.1

## Asynchronous Personal Internet Agent

### Цель проекта

Разработать рабочий прототип персонального интернет-агента для работы через асинхронные каналы связи с высокой задержкой, ограниченной пропускной способностью и нестабильной доставкой сообщений.

Система должна позволять пользователю выполнять большинство интернет-задач через передачу намерений (intent-based interaction), а не через прямое управление браузером.

Пример:

Пользователь пишет:

"Найди свежую Ubuntu Server и пришли торрент."

Вместо открытия браузера агент самостоятельно:

* анализирует задачу;
* ищет информацию;
* принимает решения;
* использует необходимые инструменты;
* возвращает результат.

---

# Основная концепция

Не проектировать удаленный браузер.

Не проектировать VNC.

Не проектировать прокси.

Основная модель:

User
→ Intent
→ Agent
→ Tools
→ Internet
→ Result

Главная цель:

максимально сократить количество обменов сообщениями между пользователем и интернетом.

# Компонент 1

Mail Gateway

Задачи:

* подключение к IMAP;
* получение новых писем;
* фильтрация отправителей;
* проверка секретного токена;
* защита от повторной обработки.

Требования:

* whitelist адресов;
* журналирование;
* обработка ошибок;
* поддержка вложений.

---

# Компонент 2

Intent Parsing Layer

Использовать PydanticAI.

Основная задача:

преобразовывать свободный текст пользователя в строго типизированные объекты задач.

Пример:

Вход:

"Скачай это видео как mp3"

Выход:

DownloadYoutubeTask(
url="...",
format="mp3"
)

---

Все задачи должны иметь строгие модели Pydantic.

Пример:

SearchTask

DownloadTask

YoutubeTask

BrowserTask

NewsTask

FileTask

---

Запрещено использовать строковые команды без типизации.

---

# Компонент 3

Workflow Engine

Использовать Burr.

Каждая задача должна выполняться как конечный автомат.

Пример:

WAITING
↓
PARSED
↓
PLANNED
↓
EXECUTING
↓
COMPLETED

или

WAITING
↓
EXECUTING
↓
FAILED

---

Состояние должно сохраняться между перезапусками процесса.

---

Каждый workflow должен сериализоваться в JSON.

# Премиальный уровень

*** Реализация плагина Firefox|Chrome для работы браузера в спецмальном APIat режиме. 

При этом большинство пользовательских задач должно решаться одним запросом и одним ответом без интерактивного веб-серфинга.

## Как компоненты ТЗ решают задачу:

* Mail Gateway + Безопасность (Компонент 1): Скрипт принимает почту по IMAP, проверяет whitelist отправителей и ищет в тексте ваш секретный токен (например, "черешня") (pp. 2-3, 7). Это полностью защищает сервер от выполнения спама или чужих писем.
* Intent Parsing Layer (Компонент 2): С помощью PydanticAI свободный текст маскировочного письма (например: "Привет! Перешли мне, пожалуйста, тот ролик про кулинарию с ютуба на этот адрес...") строго типизируется в объект DownloadYoutubeTask(url="...", format="mp3") (p. 3). Использование строковых команд запрещено, что исключает ошибки парсинга (p. 4).
* Workflow Engine (Компонент 3): Фреймворк Burr управляет жизненным циклом задачи (от WAITING до COMPLETED / FAILED) (p. 4). Поскольку Burr сохраняет и сериализует состояние автомата в JSON на диск (в SQLite), процесс может полностью завершаться после каждого шага (pp. 4, 7). Это гарантирует, что память сервера не будет утекать в долгих фоновых процессах.
* YouTube & Email Tools (Компонент 4): Утилита yt-dlp скачивает файл (p. 5), а Email Tool упаковывает его во вложения (или разбивает архив на части с помощью Archive Tool, если видео слишком тяжелое для почтового сервера) и отправляет обратно (pp. 5-6).
* Управление памятью (Chromium): Важнейший пункт вашего ТЗ — Chromium запускается только по запросу как Browser Fallback Mode (pp. 7-8). В режиме ожидания и при скачивании обычных YouTube-видео тяжелый браузер вообще не будет занимать драгоценную оперативную память (pp. 5, 7).

[Конфигурация нейросетевого ядра (LLM Provider)]
По умолчанию в качестве основного вычислительного интеллекта (ядра PydanticAI и Task Planner) комплекс использует платформу ai.wormsoft.ru. 
Все сетевые запросы к моделям должны быть строго стандартизированы под OpenAI-совместимый протокол (OpenAI API Specification). 
Для обеспечения бесшовной масштабируемости, миграции и резервного копирования, параметры подключения вынесены в изолированные переменные окружения (.env). 
Архитектура обязана поддерживать динамическую подмену провайдера «на лету» путем изменения следующих эндпоинтов в коде инициализации клиента (AsyncOpenAI / PydanticAI client):
- BASE_URL (по умолчанию: https://wormsoft.ru)
- API_KEY (локальный токен авторизации из личного кабинета)
- MODEL_NAME (целевая языковая модель для парсинга интентов, например: gpt-4o-mini или аналог)
Использование хардкода для URL-адресов или ключей внутри модулей Intent Parsing Layer запрещено.

Пример python кода интеграции:
```code
import os
from openai import AsyncOpenAI
from pydantic_ai import Agent
from pydantic_ai.models.openai import OpenAIModel

# 1. Считываем параметры из .env (по умолчанию подтягивается ai.wormsoft.ru)
BASE_URL = os.getenv("LLM_BASE_URL", "https://wormsoft.ru")
API_KEY = os.getenv("LLM_API_KEY", "your-wormsoft-token")
MODEL_NAME = os.getenv("LLM_MODEL_NAME", "gpt-4o-mini")

# 2. Инициализируем универсальный клиент
openai_client = AsyncOpenAI(base_url=BASE_URL, api_key=API_KEY)
model_backend = OpenAIModel(MODEL_NAME, openai_client=openai_client)

# 3. Передаем этот "мозг" в агент PydanticAI
intent_agent = Agent(model=model_backend, result_type=DownloadYoutubeTask)
```
